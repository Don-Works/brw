Date: 2026-06-22

Scope: validate the v0.3.0 agent-browser polish work, especially real DevTools
mobile emulation, `brw_read`, stale refs, screenshots, `brw_plan` result data,
and LLM discoverability.

## Release Verification

- `go test ./...`: passed.
- Local install: `make install-mac` copied and codesigned v0.3.0 binaries into
  `~/Library/Application Support/brw/bin`.
- Remote install: copied the same binaries/assets to `max-air`, restarted
  `co.revitt.brw.bridge`, and killed stale remote MCP wrapper children so
  mcplexer reconnected to the new binary.
- Live mcplexer catalog: `brw__brw_emulate_device` is now discoverable.
- Live smoke test: an isolated `about:blank` tab with
  `<meta name="viewport" content="width=device-width,initial-scale=1">` applied
  `device:"iphone_se"` and reported:
  - `innerWidth=375`, `innerHeight=667`
  - `devicePixelRatio=2`
  - `navigator.maxTouchPoints=5`
  - `navigator.platform="iPhone"`
  - mobile UA match true
  - `(max-width: 400px)` media query true
  The emulation was cleared and the test tab closed afterward.

## Model Review Takeaways

The most valuable finding was not about the source code. Both successful models
flagged that `brw_emulate_device` was not visible in the live mcplexer tool
surface. That was true before deployment reload: the source had the tool, but
the running remote route still used the old binary. The release process now
includes installing to `max-air`, restarting the bridge daemon, killing stale MCP
wrapper children, and reloading mcplexer.

Other recurring LLM-usability risks:

- Tool choice must be machine-readable. v0.3.0 now exposes TypeScript union
  choices for device presets, orientation, snapshot mode/format, scroll
  direction, mouse buttons, and plan/batch action names.
- Agents will miss powerful features if they are only described in prose. Keep
  new capabilities in the core profile when they unblock common workflows.
- `brw_plan` and `brw_batch` still overlap conceptually. `brw_plan` now returns
  per-step result payloads, but future work should consider unifying or more
  sharply differentiating the two surfaces.
- Dense SPA workflows still need continued stress testing around `frontier`
  limits, `mode:"all"`, stale-ref recovery, and result-size tradeoffs.
- Incognito context lifecycle is brittle for LLMs if they lose the returned
  `context_id`; a future `brw_list_contexts` would reduce leaks.
