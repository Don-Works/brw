## Features verified WORKING

- `brw_navigate_to` — reuses the tab, returns a post-nav observation. (Also confirmed via direct bridge `curl`.)
- `page_ready` alias accepted by `brw_wait_for`.
- **ID disambiguation** — Challenging DOM: 20 repeated edit/delete links → all unique, all suffixed (`e6_edit_1`, `e7_delete_1`, `e6_edit_2`…). Exactly as designed.
- **fill→type error hint** — filling a `<label>` ref returns: *"ref e3 is not fillable (label has no value property…; use brw_type… or re-run brw_snapshot to find the correct input ref)"*. Excellent.
- **JS dialog auto-dismiss** (after extension reload) — alert/confirm/prompt: click returns (~1.7s, no hang), `#result` confirms. confirm→"You clicked: Ok", prompt→"You entered: ".
- Forms end-to-end: dropdown select, checkboxes, inputs, horizontal slider (range via `brw_fill`), Form Auth login ("You logged into a secure area!").
- File upload (`brw_upload_file` → "File Uploaded!"), basic_auth + digest_auth + secure download via creds-in-URL.
- Dynamic loading/controls, notification flash, redirector, infinite scroll, jQuery menu, frames, shadow DOM, tables.
- **Token discipline**: Large & Deep DOM (2500 `<td>`) → frontier snapshot bounded to ~1 element. Semantic-first, no screenshot tokens.

## Bugs found

1. **`snapshot:true` on action tools is silently dropped on the live (upstream-http) topology.** `server.go` sets
   `WithWantSnapshot(ctx)`, but `internal/httpclient/controller.go` action methods don't forward the flag as a
   request field (only `NavigateTo` was added in this commit), and a Go context value can't cross the HTTP boundary
   to the bridge. `internal/http/server.go` *does* read `snapshot` and `bridge.go` *does* populate it — so the fix is
   ~10 lines: forward `snapshot` (from `WantSnapshotFromCtx`) in `Click`/`Fill`/`Type`/`Select`/`Press`/`Scroll`/
   `Hover`/`ClickButton`/`ClickText`/`UploadFile`. Works in single-process direct-CDP; broken on `--upstream-http`.

2. **`new_tab_id` cannot fire for `target=_blank`/`window.open`.** Confirmed: after a brw click,
   `navigator.userActivation.isActive === false`, so popups are blocked and no tab opens. Root cause is
   architectural (`bridge_mouse.go`: input is synthesized in-page as `isTrusted=false` events, "no direct CDP Input").
   Detection logic itself (bridge.go `observeActionWithBeforeAndTabs`) is correct; it just never sees a new tab.

## Limitations (extension-bridge synthetic input) — expected, worth documenting for agents

- No user activation → popups blocked (see bug 2); HTML5 drag-and-drop not triggered (drag_and_drop headers unchanged);
  document-level keydown handlers may not fire (key_presses produced no result). `brw_drag` still works for slider/pan.
- Native browser UI invisible to brw: leaked-password "Change your password" bubble (triggered here by logging in with
  the-internet's public `SuperSecretPassword!`), save-password/permission/geolocation/print dialogs, download shelf.
  CDP controls page DOM only; `brw_screenshot` is viewport-only. Agents should hand these to the human, not retry.
- `brw_console` resolves to the FOREGROUND tab (failed with an explicit `tab_id` while a `chrome://extensions` tab was
  focused) and misses load-time/uncaught errors fired before its hook injects (javascript_error → empty console).
- Minor: `brw_find({query:"username"})` ranked the `<label>` above the input (mitigated by the fill→type hint).
