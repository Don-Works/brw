## Notes
- 2026-06-14 (agent): Strategic comparison note after local Claude Chrome extension inspection (1.0.75):
  
  Where agent-browser already has an edge:
  - Harness-agnostic MCP/HTTP daemon: usable by Codex, Claude, local scripts, remote agents, and test harnesses; Claude's extension is primarily Claude-side-panel/Claude Code/native-host coupled.
  - Semantic-first control: our product rule is DOM/accessibility/readability as primary state and screenshots as fallback. Claude has a text tree but leans much harder on coordinate computer actions and screenshots.
  - Profile ownership and remote posture: our SSH/Tailscale/browser-machine model keeps authenticated Chrome profile, passkeys, downloads, and OAuth state on the trusted machine. Claude is excellent locally but product-coupled through Anthropic native host/cloud bridge.
  - Install/profile policy architecture: agent-browser already has workspace/profile/transport authorization as a product concept. Claude has managed blockedUrlPatterns/forceLoginOrgUUID, but not the same workspace-to-profile control boundary.
  - Open conformance path: we can build public fixtures and scorecards that are model/vendor independent.
  
  Ways to get ahead rather than merely match:
  - Make semantic refs better than Claude's WeakRef text refs: rerender-stable IDs, stale-ref recovery, iframe/shadow/source attribution, compact deltas, and ref-scoped snapshots.
  - Make batching better: support semantic refs, queries, coordinates, waits, assertions, downloads, network/console reads, and rollback-ish stop semantics in one structured transaction with compact observations.
  - Add safety as a differentiator: default sensitive field redaction, per-profile blocklists, action audit logs, domain drift warnings, kill switch, and explicit denied-action semantics.
  - Add testability as the product: browser automation conformance suite, replayable traces, deterministic fixtures, and perf scorecards for round trips/tokens/actions.
  - Add developer diagnostics Claude does not foreground: console/network/download/storage observations correlated to the exact tab and action version, plus DevTools MCP as a companion diagnostics plane.
  - Keep the extension bridge minimal and auditable. Claude's manifest is powerful; we should add privileges only when they buy measurable user value, with profile-policy gates.
  
  Immediate likely bug/advantage opportunity: Claude redacts password/OTP/credit-card-like values in the tree. Our current snapshot/readability code appears to expose non-hidden input values. Fixing that quickly improves both safety and credibility.
- 2026-06-14 (agent): Product constraint: preserve open-web browsing, including financial pages. Do not copy Claude-style URL category blocking as a default product behavior. Agent-browser should not decide broad categories of the web are off-limits; that is an anti-web pattern and would undermine the value of controlling the user's real browser profile.
  
  Safety should be expressed as user/workspace-owned policy and action semantics: explicit profile/workspace authorization, transparent audit logs, pause/kill switch, sensitive-value redaction, domain-drift warnings, and human handoff for MFA/CAPTCHA/fraud/consent gates. Financial sites must remain reachable and controllable when the user has authorized the profile/session, while the agent must not bypass site security controls.
- 2026-06-14 (agent): Deep-dive finding: the highest-leverage speed/quality gap is not another click primitive. It is a Playwright-grade semantic action layer for agents.
  
  Claude has: browser_batch, coordinate computer actions, console/network reads, tab groups, workflow recording, permission UI, and polished side-panel/native-host integration. It appears to use CDP Network/Runtime/Input/Page events, but its page tree is still a per-call content-script text tree with WeakRef refs, and its primary action model is still coordinate-heavy.
  
  Agent-browser has: better daemon/MCP architecture, semantic refs, profile-policy/remote-control posture, compact action observations, event-driven waits, hover/evaluate/network/plan now present in local code, and a roadmap for page kernel/events.
  
  Killer features neither side fully has yet:
  1. Playwright-grade actionability + auto-wait for agent actions: before click/fill/drag, wait for exactly-one target, visible, stable box, receives events, enabled/editable where relevant, then fail with structured reason. Playwright documents this as core to reliability; it is missing as a first-class semantic contract in both Claude and agent-browser.
  2. Web-first assertions as browser tools: expect role/text/value/count/url/network/console/download conditions that auto-retry and return compact evidence. This turns scripts from click-sleep-snapshot into click-expect-observe.
  3. Page kernel + semantic event journal: inject once per document, maintain refs/boxes/parts/focus/live/network state, drain deltas instead of rescanning. This is the big speed jump: fewer CDP round trips and fewer model tokens.
  4. Intent engine / locator ranking with reasons: BM25/trigram over label, role, placeholder, form owner, aria relations, nearby text, URL context, history, and per-origin affordance cache. Agents ask for “postcode field” or “checkout button” and get ranked targets with confidence/reasons.
  5. Form lens and task lenses: when inside a form/table/list/cart/checkout, return the relevant schema + invalid fields + submit controls + live messages, not the whole page. This should beat Claude’s broad read_page and our current frontier on real workflows.
  6. Visual islands: automatically detect canvas/maps/charts/image-only widgets and return cropped screenshot/OCR only for that island. Keep normal DOM pages semantic-first.
  7. Performance scoreboard: every browser tool should report hidden timing/bytes/round-trip counters into a trace so we can optimize median task time, not just individual tool latency.
  
  Strategic push: build these as vendor-neutral MCP primitives. Do not just clone Claude's side-panel behavior; leapfrog it with deterministic, observable, low-token browser transactions.
- 2026-06-14 (agent): Architecture principle: make the browser extension/page kernel/daemon as compos mentis as possible. Any deterministic browser work that can happen locally in Chrome is faster than a model round trip and costs zero tokens. Prefer local computation for actionability checks, target ranking, stale-ref recovery, semantic diffs, form lenses, assertions, waits, event journals, scroll/container selection, and batch execution.
  
  The model should provide intent, approve ambiguous/high-risk choices, and inspect compact evidence. It should not repeatedly re-read full page state or reason through mechanics the browser runtime can compute exactly.
  
  Acceptance criteria for future browser work:
  - Every new model-visible tool asks: can the extension/page kernel do this internally and return a compact result?
  - Avoid exposing raw state when a local ranked/filtered/validated answer is possible.
  - Prefer one local transaction over multiple model/tool turns.
  - Local intelligence must be deterministic, inspectable, and safety-gated; no hidden site-security bypass logic.
  - Performance metrics should count model round trips and output tokens avoided, not just wall-clock latency.
